# L2.1 — conservation flags match the oracle

**Test:** `test_L2_conservation_matches_oracle[<scenario>]` (parametrised ×12)
**Layer:** SPARQL rule + verdict · **Status:** GREEN (built)

## What it checks
After running `check-mass-conservation.rq`, the set of wholes stamped
`futuram:massConserved false` **exactly equals** the set the oracle says
overshoot (minimum fraction sum > 1.0 kg/kg).

## Checked against
`SupplyChain.conservation()` — the Python ground truth, which for each whole
computes the min/max sum of its direct part fractions (normalising g/kg ×0.001)
and marks `overshoot = (min_sum > 1.0)`.

## Expected outcome per scenario
| scenario | min sum | flagged? |
|----------|---------|----------|
| 02 shortfall | 0.30 | no |
| 03 exact_one | 1.00 | no (boundary) |
| 04 overshoot | 1.20 | **yes** |
| 05 mixed_units | 0.21 | no (after g/kg scaling) |
| others | <1 | no |

The flagged set must equal the oracle's overshoot set for **every** scenario.

## Why it matters
Verifies the SPARQL SUM + GROUP BY + the g/kg→kg/kg normalisation across all
chain shapes (shortfall, exact boundary, overshoot, mixed units, nested,
multipath). The boundary (exactly 1.0 not flagged) and the mixed-unit
normalisation are the subtle cases.

that is a one-line oracle + rule change — flag it if so.)
